# Архитектурные решения SemanticNews

## Как пользоваться файлом

Это короткий чек-лист принятых архитектурных решений, разбитый по этапам плана разработки.

Правило работы:

- перед началом очередного этапа сначала смотреть только соответствующий раздел этого файла;
- если нужны детали по потоку данных или структуре папок, потом открывать `TECHNICAL_CONTEXT.md` и `PROJECT_STRUCTURE.md`;
- если решение меняется по согласованию, сначала обновлять этот файл, а затем остальные документы.

## 1. Модель данных и SQLite

- `SQLite` — основной источник правды.
- Основные сущности: `Source`, `Article`, `Request`, `SearchResult`, служебные логи и два справочника типов.
- Связи в реализации строятся через суррогатные `id`.
- Таблицы `Request` и `SearchResult` остаются в схеме БД.
- В схему добавляются справочники `source_type` и `article_type`.
- `Source` получает поле `source_type_id`.
- `Article` получает поле `article_type_id`.
- `source_type` задается из фиксированного списка значений, а не вводится свободным текстом.
- Для пользовательских источников тип источника выбирается пользователем из выпадающего списка.
- `article_type` трактуется как тип материала по форме представления и каналу публикации, а не по смыслу текста.
- Для `article_type` используется фиксированный список значений, включая `other`.
- Значение `other` допустимо и не нарушает нормализацию, потому что остается ссылкой на запись справочника, а не свободным текстовым исключением.
- Статья без текста не сохраняется.
- Если дата публикации отсутствует, подставляется текущее время.
- Дубли определяются по URL статьи.
- Дубли повторно не сохраняются и отдельно не логируются.
- Источники хранятся в БД.
- Стартовые источники добавляются seed-скриптом.
- В служебном справочнике событий используется таблица `EventType`, а не `Error`.
- В `EventType` поле `code` остается уникальным бизнес-кодом события, а `id` используется как технический суррогатный ключ для связей.
- Поле `description` в справочниках нужно для человекочитаемого пояснения значения в UI, админских формах и документации.
- У `Source.name` поле обязательное; если имя не удалось определить автоматически, подставляется дефолтное значение вроде `unknown`.
- У `Source` добавляется булево поле активности источника.
- Для `SearchResult` фиксируется ограничение `UNIQUE(request_id, position)`.

## 2. Контракт parser -> service -> repository

- Парсер не работает с ORM и БД напрямую.
- Цепочка слоев: `parser -> DTO -> ingestion_service -> repository`.
- DTO используется для передачи нормализованных данных между слоями.
- ORM-модели используются только для работы с SQLite.
- Репозиторий отвечает за CRUD и ORM-запросы, но не за бизнес-валидацию сценария.
- Ожидаемые прикладные проверки, такие как дубли, пустой текст и подстановка даты, выполняются в сервисе до вызова репозитория.
- Если база данных все же выбрасывает ошибку ограничения, транзакция откатывается на уровне ORM-сессии, а решение о реакции на ошибку принимается в сервисе.
- Репозитории не должны содержать пользовательский поиск, работу с FAISS и другую бизнес-логику поверх данных.

## 2.1. Правила работы с ORM-сессией

- Одна операция или один сценарий использует свою отдельную ORM-сессию.
- Для чтения используется `get_session()`.
- Для записи используется `session_scope()`.
- Сессии не делятся между независимыми сценариями и потоками.
- ORM-объекты, возвращенные из репозитория после закрытия сессии, безопасно использовать для уже загруженных простых полей, но не стоит рассчитывать на ленивую подгрузку связанных сущностей вне активной сессии.

## 3. Ingestion pipeline

- Базовый pipeline: получить ссылки -> скачать статьи -> отфильтровать пустые -> сохранить в SQLite -> обновить FAISS -> записать финальный статус.
- Во время разработки ingestion можно запускать из скрипта.
- В приложении ingestion должен быть доступен из UI.
- Позже допустим автозапуск при первом старте, если БД пустая.
- Периодический автосбор можно оставить как позднее улучшение.
- В ближайшем MVP реально поддерживаются HTML-страницы и текущий сценарий sitemap/html.
- Рабочий parser package живет внутри `project/app/parsers`; корневой `parcer.py` считается удаленным legacy-слоем и не должен возвращаться.
- Parser API должен оставаться typed: внутренние parser-модули возвращают структурированные модели, а не сырые `dict`.
- Для текста статьи основным извлекателем считается `trafilatura`, для даты публикации — `htmldate` с дополнением через meta/time/JSON-LD, а `BeautifulSoup` используется как вспомогательный HTML-разборщик.
- Локальные проверки parser-модулей нужно выполнять через `.\.venv\Scripts\python.exe`.
- Поддержка Telegram, PDF, презентаций и других форматов закладывается архитектурно, но реализуется позже.

## 4. Фоновый ingestion

- Ingestion не должен блокировать UI.
- На текущем стеке делаем фоновую задачу, а не смену backend-стека.
- Для первой версии статус фоновой задачи хранится в памяти Flask-процесса.
- Отдельную таблицу под статус не добавляем.
- Это решение рассчитано на локальное приложение с одним пользователем и одной активной задачей.

## 5. Базовый MVC и новости

- Базовый vertical slice: список новостей и карточка статьи.
- `news_controller` принимает запрос и вызывает сервис.
- `news_service` готовит данные для интерфейса.
- `news_repository` читает статьи из SQLite.
- Шаблоны только отображают данные, без бизнес-логики.

## 6. UI и статус ingestion

- У ingestion будет UI-блок или экран.
- Статус показывается через JavaScript polling status endpoint.
- `ingestion_controller` может отдавать JSON со статусом задачи.
- Логика самой загрузки остается в `ingestion_service`.

## 7. Источники

- Источники задаются через БД.
- Есть стартовый набор источников “из коробки”.
- Пользователь в будущем сможет добавить свой источник.
- Отдельный `source_service` нужен, но не входит в ближайший MVP.
- Тип источника хранится в справочнике `source_type`.
- Минимальный стартовый набор `source_type`: `news_media`, `organization_site`, `personal_page`, `telegram_channel`.

## 8. Эмбеддинги и FAISS

- `FAISS` — отдельный индекс, а не БД.
- Связь `FAISS <-> SQLite` идет через `article_id`.
- Сначала статьи сохраняются в `SQLite`, потом отдельным шагом обновляется `FAISS`.
- После ingestion индексируются только новые статьи.
- Полный rebuild не основной сценарий, но допустим как служебная возможность.
- Логика индекса живет в `indexing_service`.

## 8.1. Типы материалов и расширяемость парсеров

- `article_type` определяется ingestion-логикой, а не пользователем.
- `article_type` должен отражать формат материала или канал публикации.
- Минимальный стартовый набор `article_type`: `web_article`, `telegram_post`, `pdf_document`, `presentation`, `other`.
- Если тип материала не удалось определить, используется `other`.
- Фильтрация по типам материалов нужна в будущем, но не входит в ближайший MVC.

## 9. Search architecture

- `embedding_service` строит эмбеддинги статей и запросов.
- `indexing_service` работает с `FAISS`.
- `search_service` orchestrates поиск целиком.
- Контроллеры не должны напрямую работать с FAISS.
- Каждый поисковый запрос сохраняется в `Request`.
- Top-k результаты сохраняются в `SearchResult`.

## 10. ML-этап

- Классификации не будет.
- ML-цель — адаптация embedding-модели под новостной домен.
- Базовая модель: `sentence-transformers/all-MiniLM-L6-v2`.
- Основная идея датасета: пары `заголовок -> текст статьи`.
- После смены модели требуется переиндексация `FAISS`.
- Если локальная дообученная модель есть, приложение использует ее.
- Если ее нет, используется базовая модель.

## 11. Desktop и финализация

- Desktop-оболочка остается на `pywebview`.
- Сначала доводится рабочий web/MVC-контур.
- Потом он проверяется внутри `pywebview`.
- Упаковка в `.exe` идет только после стабилизации основного контура.
